## 1. Portal Configuration Object

A Configuration controls the features and appearance of the Billing Portal. You can create multiple configurations for different customer segments.

| Parameter | Type | Description |
|---|---|---|
| `id` | string | Unique identifier for the Configuration. |
| `active` | boolean | Whether this configuration can be used to create portal sessions. |
| `business_profile` | object | Branding and legal information displayed in the portal. |
| `default_return_url` | string, nullable | Default URL to redirect customers back to your app. |
| `features` | object | Feature settings controlling what customers can do in the portal. |
| `is_default` | boolean | Whether this is the default configuration for your account. |
| `login_page` | object | Settings for the portal login page. |
| `metadata` | object, nullable | Key-value pairs for storing additional information. |
| `name` | string, nullable | A human-readable name for the configuration (max 256 chars). |

### business_profile

| Field | Type | Description |
|---|---|---|
| `headline` | string | Headline displayed in the portal header. |
| `privacy_policy_url` | string | Link to your privacy policy. |
| `terms_of_service_url` | string | Link to your terms of service. |

### features

| Feature | Sub-fields | Description |
|---|---|---|
| `customer_update` | `allowed_updates` (array: `email`, `tax_id`), `enabled` | Allow customers to update their profile information. |
| `invoice_history` | `enabled` | Allow customers to view and download past invoices. |
| `payment_method_update` | `enabled` | Allow customers to add, remove, or update payment methods. |
| `subscription_cancel` | `enabled`, `mode` (`immediately` / `at_period_end`), `cancellation_reason` (object), `proration_behavior` | Allow customers to cancel subscriptions. |
| `subscription_update` | `enabled`, `default_allowed_updates` (array: `price`, `quantity`, `trial_end`, `metadata`), `proration_behavior` | Allow customers to change subscription plans or quantities. |

### subscription_cancel.cancellation_reason

| Field | Type | Description |
|---|---|---|
| `enabled` | boolean | Whether to prompt for a cancellation reason. |
| `options` | array | Reason options: `too_expensive`, `missing_features`, `switched_service`, `unused`, `other`. |

### login_page

| Field | Type | Description |
|---|---|---|
| `enabled` | boolean | Whether the login page is enabled. |
| `url` | string | The URL of the login page (read-only, generated by Stripe). |

> **Gap:** The Configuration object does not support per-product cancellation or update rules. All subscription features apply uniformly to every subscription the customer has. Use `flow_data` on the Session to direct customers to a specific subscription.

## 2. Create a Configuration

```
POST /v1/billing_portal/configurations
```

### Required Parameters

| Parameter | Type | Description |
|---|---|---|
| `features` | object | Feature settings for the portal. At least one feature must be enabled. |

### Optional Parameters

| Parameter | Type | Description |
|---|---|---|
| `business_profile` | object | Branding and legal links. |
| `default_return_url` | string | URL to return customers to after portal use. |
| `login_page` | object | Login page settings. |
| `metadata` | object | Key-value pairs. |
| `name` | string | Human-readable name (max 256 chars). |

### Example: Create a B2B Configuration with Cancellation Reasons

```bash
curl https://api.stripe.com/v1/billing_portal/configurations \
  -u sk_test_xxx: \
  -d "features[subscription_cancel][enabled]"=true \
  -d "features[subscription_cancel][mode]"=at_period_end \
  -d "features[subscription_cancel][cancellation_reason][enabled]"=true \
  -d "features[subscription_cancel][cancellation_reason][options][]"=too_expensive \
  -d "features[subscription_cancel][cancellation_reason][options][]"=missing_features \
  -d "features[subscription_cancel][cancellation_reason][options][]"=switched_service \
  -d "features[subscription_cancel][cancellation_reason][options][]"=unused \
  -d "features[subscription_cancel][cancellation_reason][options][]"=other \
  -d "features[subscription_update][enabled]"=true \
  -d "features[subscription_update][default_allowed_updates][]"=price \
  -d "features[subscription_update][default_allowed_updates][]"=quantity \
  -d "features[subscription_update][proration_behavior]"=create_prorations \
  -d "features[payment_method_update][enabled]"=true \
  -d "features[invoice_history][enabled]"=true \
  -d "features[customer_update][enabled]"=true \
  -d "features[customer_update][allowed_updates][]"=email \
  -d "features[customer_update][allowed_updates][]"=tax_id \
  -d "business_profile[headline]"="Manage your subscription" \
  -d default_return_url="https://app.example.com/billing" \
  -d name="B2B Growth Tier Portal"
```

## 3. Update a Configuration

```
POST /v1/billing_portal/configurations/:id
```

Update an existing configuration. All fields from Create are updatable. Set `active=false` to deactivate a configuration without deleting it.

### Example: Disable subscription cancellation

```bash
curl https://api.stripe.com/v1/billing_portal/configurations/bpc_xxx \
  -u sk_test_xxx: \
  -d "features[subscription_cancel][enabled]"=false
```

> **Gap:** Updating a configuration does not affect portal sessions already in progress. Changes only apply to sessions created after the update.

## 4. Retrieve a Configuration

```
GET /v1/billing_portal/configurations/:id
```

Retrieve an existing configuration by its ID.

```bash
curl https://api.stripe.com/v1/billing_portal/configurations/bpc_xxx \
  -u sk_test_xxx:
```

## 5. List Configurations

```
GET /v1/billing_portal/configurations
```

Returns a paginated list of all portal configurations. Supports `active` and `is_default` filters.

```bash
curl https://api.stripe.com/v1/billing_portal/configurations \
  -u sk_test_xxx: \
  -d active=true \
  -d limit=10
```

## 6. Portal Session Object

A Portal Session represents a single visit to the Billing Portal. Sessions are short-lived and single-use.

| Parameter | Type | Description |
|---|---|---|
| `id` | string | Unique identifier for the Session. |
| `configuration` | string | The Configuration used for this Session. |
| `created` | timestamp | Time at which the Session was created. |
| `customer` | string | The Customer this Session is for. |
| `customer_account` | string | The customer account (Connect platforms). |
| `flow` | object, nullable | Information about the flow the customer is directed to. |
| `locale` | enum, nullable | The locale for the portal. Supports 45+ locales. |
| `on_behalf_of` | string | The connected account (Connect platforms). |
| `return_url` | string, nullable | URL the customer is redirected to when they are done in the portal. |
| `url` | string | The short-lived, single-use URL for the portal session. |

### flow

| Field | Type | Description |
|---|---|---|
| `type` | enum | The flow type. One of `payment_method_update`, `subscription_cancel`, `subscription_update`, `subscription_update_confirm`. |
| `after_completion` | object | Behavior after the flow is completed. |

> **Gap:** Portal Sessions cannot be retrieved or updated after creation. The `url` is single-use and short-lived. If the customer needs a new link, create a new Session.

## 7. Create a Portal Session

```
POST /v1/billing_portal/sessions
```

### Required Parameters

| Parameter | Type | Description |
|---|---|---|
| `customer` | string | The ID of the Customer for this Session. |

### Optional Parameters

| Parameter | Type | Description |
|---|---|---|
| `configuration` | string | The Configuration to use. Defaults to the default configuration. |
| `customer_account` | string | The customer account (Connect platforms). |
| `flow_data` | object | Direct the customer to a specific flow. |
| `locale` | enum | Portal locale (45+ options). |
| `on_behalf_of` | string | The connected account (Connect platforms). |
| `return_url` | string | URL to redirect the customer when they are done. Overrides `default_return_url`. |

### flow_data types

| Type | Description |
|---|---|
| `payment_method_update` | Direct customer to update their payment method. |
| `subscription_cancel` | Direct customer to cancel a specific subscription. |
| `subscription_update` | Direct customer to update a specific subscription. |
| `subscription_update_confirm` | Direct customer to confirm a specific subscription update. |

### Example: Create a Portal Session with deep link to cancellation

```bash
curl https://api.stripe.com/v1/billing_portal/sessions \
  -u sk_test_xxx: \
  -d customer=cus_abc123 \
  -d configuration=bpc_xxx \
  -d "flow_data[type]"=subscription_cancel \
  -d "flow_data[subscription_cancel][subscription]"=sub_xyz789 \
  -d return_url="https://app.example.com/billing"
```

### Example: Basic portal session

```bash
curl https://api.stripe.com/v1/billing_portal/sessions \
  -u sk_test_xxx: \
  -d customer=cus_abc123 \
  -d return_url="https://app.example.com/billing"
```

## 8. B2B SaaS Patterns

### Subscription Cancel with Reason Capture for Churn Analysis

Enable `cancellation_reason` on your configuration to collect structured feedback when customers cancel. Use the `customer.subscription.deleted` webhook to read `cancellation_details.reason` and `cancellation_details.feedback` for churn reporting.

```bash
# Configuration with cancellation reasons enabled
curl https://api.stripe.com/v1/billing_portal/configurations \
  -u sk_test_xxx: \
  -d "features[subscription_cancel][enabled]"=true \
  -d "features[subscription_cancel][mode]"=at_period_end \
  -d "features[subscription_cancel][cancellation_reason][enabled]"=true \
  -d "features[subscription_cancel][cancellation_reason][options][]"=too_expensive \
  -d "features[subscription_cancel][cancellation_reason][options][]"=missing_features \
  -d "features[subscription_cancel][cancellation_reason][options][]"=switched_service \
  -d "features[subscription_cancel][cancellation_reason][options][]"=unused \
  -d "features[subscription_cancel][cancellation_reason][options][]"=other
```

> **Gap:** Cancellation reasons from the portal are stored on the Subscription object under `cancellation_details`, not on the portal Session. There is no dedicated webhook event for the reason itself — read it from the subscription update or deletion event.

### Plan Upgrades and Downgrades with Proration

Enable `subscription_update` with `proration_behavior` set to `create_prorations` for immediate billing adjustments when customers change plans.

```bash
curl https://api.stripe.com/v1/billing_portal/configurations/bpc_xxx \
  -u sk_test_xxx: \
  -d "features[subscription_update][enabled]"=true \
  -d "features[subscription_update][default_allowed_updates][]"=price \
  -d "features[subscription_update][default_allowed_updates][]"=quantity \
  -d "features[subscription_update][proration_behavior]"=create_prorations
```

### Payment Method Self-Service to Reduce Failed Payments

Enable `payment_method_update` to let customers fix their payment methods before dunning escalates. Combine with Stripe's Smart Retries for maximum recovery.

```bash
# Deep link to payment method update
curl https://api.stripe.com/v1/billing_portal/sessions \
  -u sk_test_xxx: \
  -d customer=cus_abc123 \
  -d "flow_data[type]"=payment_method_update \
  -d return_url="https://app.example.com/billing"
```

### Deep Linking via flow_data

Use `flow_data` to send customers directly to a specific action rather than the portal home page. This is useful for in-app prompts like "Update your payment method" or "Cancel your subscription."

### Multiple Configurations for Different Client Tiers

Create separate configurations for different customer segments. For example:
- **Free tier:** Invoice history only, no self-service cancel.
- **Growth tier:** Full self-service including plan changes and cancellation with reasons.
- **Enterprise tier:** Payment method updates and invoice history only (cancellation handled by account management).

```bash
# Create session with tier-specific configuration
curl https://api.stripe.com/v1/billing_portal/sessions \
  -u sk_test_xxx: \
  -d customer=cus_abc123 \
  -d configuration=bpc_enterprise_config \
  -d return_url="https://app.example.com/billing"
```

> **Gap:** There is no way to dynamically select a configuration based on the customer's subscription or metadata. Your application must determine the correct configuration ID before creating the portal session.
